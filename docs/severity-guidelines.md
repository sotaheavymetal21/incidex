# インシデントSeverity（重要度）設定ガイドライン

> **最終更新**: 2025-12-25

## 概要

インシデントのSeverity（重要度）は、対応の優先順位、エスカレーションパス、通知対象を決定する最も重要な指標です。正確な重要度設定により、効率的なリソース配分と迅速な問題解決が可能になります。

---

## Severity分類基準

### 🔴 Critical (最重要)

**定義**: サービス全体の停止、または主要機能が完全に利用できない状態

**影響範囲**:
- 全ユーザーまたは大多数のユーザーに影響
- サービスの主要機能が完全に停止
- ビジネスに深刻な金銭的損失または評判の毀損が発生

**対応目標**:
- **検知から15分以内**に対応開始
- 即座にオンコール担当者をページング
- 関係者全員に即座通知（SMS/電話含む）

**具体例**:
- ✗ データベース全体がダウン
- ✗ 認証システムの完全な障害（全ユーザーがログイン不可）
- ✗ 本番環境の完全なサービス停止
- ✗ 重大なセキュリティ侵害（進行中）
- ✗ データ損失の発生または高リスク
- ✗ 決済システムの完全停止
- ✗ 主要APIの完全障害（エラー率 > 50%）

**判断基準チェックリスト**:
- [ ] ユーザーがサービスの主要機能を**まったく使用できない**
- [ ] 回避策が**存在しない**
- [ ] ビジネスへの**深刻な財務的影響**がある（時給数百万円以上の損失）
- [ ] メディアや顧客からの**大規模なクレーム**が予想される

---

### 🟠 High (高)

**定義**: 主要機能の一部が機能せず、回避策がない状態

**影響範囲**:
- 大規模なユーザーグループに影響（全体の10%以上）
- 主要機能の重大な劣化
- ビジネスに顕著な影響

**対応目標**:
- **1時間以内**に対応開始
- 担当チームに緊急通知
- 定期的なステータス更新（30分ごと）

**具体例**:
- ⚠ 決済システムの一部でエラー発生（一部のカードが処理できない）
- ⚠ 特定地域・デバイスでのサービス停止
- ⚠ 重要なAPIのエラー率が20-50%
- ⚠ パフォーマンスの著しい低下（レスポンスタイム10倍以上）
- ⚠ 一部機能の完全停止（ただし回避策なし）
- ⚠ セキュリティ脆弱性の発見（既知の攻撃パターン）
- ⚠ データの不整合（重要なレコードに影響）

**判断基準チェックリスト**:
- [ ] 多数のユーザーが**重要な機能を使用できない**
- [ ] 回避策が**ない、または非常に煩雑**
- [ ] ビジネスへの**顕著な財務的影響**がある（時給数十万円の損失）
- [ ] SLA違反が**発生または差し迫っている**

---

### 🟡 Medium (中)

**定義**: 機能が劣化しているが、回避策が存在する状態

**影響範囲**:
- 限定的なユーザーグループに影響（全体の10%未満）
- 機能は利用可能だが劣化している
- ビジネスへの影響は中程度

**対応目標**:
- **4時間以内**に対応開始
- 通常の業務時間内で優先的に対応
- 日次でのステータス更新

**具体例**:
- ⚡ 一部機能の遅延（使用は可能だが遅い）
- ⚡ 非重要な機能の障害（例: レポート生成が失敗）
- ⚡ 軽微なパフォーマンス低下（レスポンスタイム2-3倍）
- ⚡ 一部APIのエラー率が5-20%
- ⚡ 一部ユーザーのみに影響する問題（特定の設定・環境）
- ⚡ バックアップシステムの障害（本番は稼働中）
- ⚡ 監視・ログシステムの部分的障害

**判断基準チェックリスト**:
- [ ] ユーザーは**回避策で作業を継続できる**
- [ ] 影響範囲が**限定的**
- [ ] ビジネスへの影響は**管理可能**（時給数万円程度の損失）
- [ ] 緊急性は**高くないが無視できない**

---

### 🟢 Low (低)

**定義**: 軽微な問題で、ユーザー体験への影響が最小限

**影響範囲**:
- 個別ユーザーまたは内部のみに影響
- 機能的な問題はほぼなし
- ビジネスへの影響が極めて小さい

**対応目標**:
- **通常の業務時間内**で対応
- 次回のスプリント/リリースで修正を検討
- 必要に応じてバックログに追加

**具体例**:
- 🔵 UIの軽微な不具合（表示のズレ、スタイルの問題）
- 🔵 誤字脱字、ドキュメントの軽微な誤り
- 🔵 ログメッセージの問題
- 🔵 開発環境のみの問題
- 🔵 既知の制限事項に関する問い合わせ
- 🔵 機能改善要望（現状でも使用可能）
- 🔵 内部ツールの軽微な不具合

**判断基準チェックリスト**:
- [ ] ユーザーへの影響が**ほぼない**
- [ ] ビジネスへの影響が**無視できるレベル**
- [ ] 即座の対応が**不要**
- [ ] 計画的な修正で**十分対応可能**

## Severityの変更

インシデントの調査が進むにつれて、Severityを適切に調整してください:

### アップグレード（重要度を上げる）すべきケース:
- 影響範囲が当初の想定より広かった
- 回避策が機能しないことが判明
- ビジネスへの影響が深刻であることが判明
- 根本原因がより重大な問題である

### ダウングレード（重要度を下げる）すべきケース:
- 有効な回避策が見つかった
- 影響範囲が当初の想定より限定的だった
- 暫定的な修正が適用された
- 誤検知だった

---

## ベストプラクティス

### ✅ 推奨される運用:

1. **疑わしい場合は高めに設定**: 後でダウングレードする方が安全
2. **ビジネス影響を最優先**: 技術的複雑さより、ユーザー/ビジネスへの影響で判断
3. **定期的な見直し**: 状況変化に応じてSeverityを更新
4. **明確なコミュニケーション**: Severity変更時は理由を記録
5. **チーム合意**: 曖昧な場合はチームで議論して決定

### ❌ 避けるべき運用:

1. **過小評価**: 問題を小さく見せようとする
2. **過大評価の常習化**: すべてをCriticalにすると、本当の緊急時に信頼を失う
3. **技術的興味で判断**: 技術的に面白い問題でも、影響が小さければLow
4. **放置**: 状況が変化してもSeverityを更新しない
5. **主観的判断のみ**: データや影響範囲を確認せずに設定

## SLA対応時間の目安

| Severity | 初動対応 | 暫定対応 | 恒久対応 | 報告頻度 |
|----------|---------|---------|---------|----------|
| **Critical** | 15分以内 | 1時間以内 | 24時間以内 | 30分ごと |
| **High** | 1時間以内 | 4時間以内 | 1週間以内 | 1時間ごと |
| **Medium** | 4時間以内 | 1営業日以内 | 2週間以内 | 1日1回 |
| **Low** | 1営業日以内 | 1週間以内 | 次回リリース | 必要に応じて |
